## Overview

### What is State Synchronization?

State synchronization ensures that **two representations of the same data remain consistent**. In our case:

- **FalkorDB (Source of Truth)**: Stores the code property graph (nodes and edges)
- **Webview (Visualization)**: Displays the graph to the user

**The Challenge**: These two systems can become **out of sync** when:
1. The database is updated but the view isn't notified
2. External changes occur when the extension isn't running
3. Errors occur during update operations
4. The user kills the IDE mid-operation

---

## The Problem

### Scenario 1: External File System Changes

```
1. User opens VS Code with extension active
2. Extension indexes workspace → Creates nodes in FalkorDB
3. User closes VS Code (extension deactivates, file watchers stop)
4. User deletes files using terminal: `rm -rf src/components/`
5. User reopens VS Code
6. Problem: FalkorDB still has nodes for deleted files!
   - Database shows files that don't exist
   - Graph visualization is incorrect
```

### Scenario 2: Failed Update Operations

```
1. File is created → Extension detects change
2. Database write succeeds → Node created
3. Network issue → Graph view refresh fails
4. Problem: Database updated but view shows stale data
```

### Scenario 3: Large-Scale Changes

```
1. User runs `git checkout feature-branch`
2. 500 files changed in 2 seconds
3. File watcher triggers 500 events
4. Problem: How do we update efficiently without freezing UI?
```

## Solution Architecture

### Pattern: Stale-While-Revalidate

This pattern is used by:
- HTTP caching (Cache-Control: stale-while-revalidate)
- VS Code's symbol indexing
- Git status in editors

**Core Idea**:
1. **Show cached data immediately** (user sees something fast)
2. **Validate in background** (ensure correctness without blocking)
3. **Update when ready** (apply corrections gracefully)

### Flow Diagram

```
Extension Activation
       ↓
┌─────────────────────────────────────────────────────┐
│  PHASE 1: FAST LOAD (< 100ms)                       │
│  - Load graph data from FalkorDB                    │
│  - Display in webview immediately                   │
│  - Show "Validating..." indicator                   │
└─────────────────────────────────────────────────────┘
       ↓
┌─────────────────────────────────────────────────────┐
│  PHASE 2: BACKGROUND RECONCILIATION (async)         │
│  - Check if DB nodes match file system              │
│  - Find orphaned nodes (files deleted externally)   │
│  - Find missing nodes (files added externally)      │
│  - Update database with corrections                 │
│  - Refresh graph view                               │
│  - Clear "Validating..." indicator                  │
└─────────────────────────────────────────────────────┘
       ↓
┌─────────────────────────────────────────────────────┐
│  PHASE 3: CONTINUOUS SYNC                           │
│  - File watchers handle real-time changes           │
│  - Periodic reconciliation (every 5 minutes)        │
│  - Git operation detection → trigger reconciliation │
└─────────────────────────────────────────────────────┘
```

## Core Components

### 1. GraphSynchronizer Service

**Responsibilities**:
- Load graph data on startup
- Reconcile file system with database
- Handle periodic validation
- Coordinate updates between DB and view

**Key Methods**:
```typescript
class GraphSynchronizer {
  // Load data from DB and display immediately
  async loadGraphFromDatabase(): Promise<void>

  // Reconcile DB with file system in background
  async reconcileInBackground(): Promise<void>

  // Find nodes in DB where files no longer exist
  async findOrphanNodes(): Promise<string[]>

  // Find files that exist but aren't in DB
  async findMissingFiles(): Promise<string[]>

  // Clean up orphaned nodes
  async cleanupOrphans(nodeIds: string[]): Promise<void>

  // Start periodic reconciliation timer
  startPeriodicReconciliation(): void
}
```

### 2. Transaction Manager

**Responsibilities**:
- Ensure DB writes and view updates happen atomically
- Rollback on failures
- Prevent partial updates

**Key Concept**: If DB write succeeds but view update fails, we have inconsistency. Solution: Use a two-phase update pattern.

### 3. Change Detection

**Responsibilities**:
- Efficiently detect which files have changed
- Minimize file system scanning
- Use timestamps, hashes, or metadata

## Implementation Guide

### Step 1: Create GraphSynchronizer Service

Create `src/services/GraphSynchronizer.ts`:

```typescript
import * as fs from 'fs';
import * as path from 'path';
import * as vscode from 'vscode';
import { FalkorDBService } from './FalkorDBService';
import { GraphViewProvider } from '../providers/GraphViewProvider';

export class GraphSynchronizer {
	private dbService: FalkorDBService;
	private graphView?: GraphViewProvider;
	private isReconciling = false;
	private lastReconciliation = 0;
	private reconciliationTimer?: NodeJS.Timeout;
	private workspaceRoot: string;

	constructor() {
		this.dbService = FalkorDBService.getInstance();
		this.workspaceRoot = vscode.workspace.workspaceFolders?.[0]?.uri.fsPath || '';
	}

	/**
	 * Set the graph view provider for updates
	 */
	public setGraphView(graphView: GraphViewProvider): void {
		this.graphView = graphView;
	}

	/**
	 * PHASE 1: Load graph data immediately on startup
	 * Goal: Show user something within 100ms
	 */
	public async loadGraphFromDatabase(): Promise<void> {
		try {
			// Connect to database
			await this.dbService.connect();

			// Get all nodes and edges (from cached DB)
			const { nodes, edges } = await this.dbService.getAllNodesAndEdges();

			// Update graph view immediately (stale data is OK)
			if (this.graphView) {
				await this.graphView.refresh();
			}

			console.log(`Loaded ${nodes.length} nodes, ${edges.length} edges from database`);
		} catch (error: any) {
			console.error('Failed to load graph from database:', error);
			vscode.window.showWarningMessage('Could not load code graph. Will rebuild.');
		}
	}

	/**
	 * PHASE 2: Reconcile database with file system in background
	 * Goal: Ensure DB matches reality without blocking startup
	 */
	public async reconcileInBackground(): Promise<void> {
		// Don't run multiple reconciliations simultaneously
		if (this.isReconciling) {
			console.log('Reconciliation already in progress, skipping');
			return;
		}

		this.isReconciling = true;

		// Show progress indicator
		await vscode.window.withProgress({
			location: vscode.ProgressLocation.Notification,
			title: 'Validating code graph...',
			cancellable: false
		}, async (progress) => {
			try {
				// Step 1: Find orphaned nodes (files deleted externally)
				progress.report({ message: 'Checking for deleted files...' });
				const orphans = await this.findOrphanNodes();

				if (orphans.length > 0) {
					progress.report({ message: `Cleaning up ${orphans.length} deleted files...` });
					await this.cleanupOrphans(orphans);
				}

				// Step 2: Find missing nodes (files added externally)
				progress.report({ message: 'Checking for new files...' });
				const missingFiles = await this.findMissingFiles();

				if (missingFiles.length > 0) {
					progress.report({ message: `Indexing ${missingFiles.length} new files...` });
					await this.indexMissingFiles(missingFiles);
				}

				// Step 3: Verify metadata is up-to-date
				progress.report({ message: 'Verifying file metadata...' });
				await this.verifyFileMetadata();

				// Update last reconciliation time
				this.lastReconciliation = Date.now();
				await this.dbService.query(
					`MERGE (m:Metadata {key: 'lastReconciliation'}) SET m.value = $time`,
					{ time: this.lastReconciliation }
				);

				// Refresh view with corrected data
				if (this.graphView) {
					await this.graphView.refresh();
				}

				const totalChanges = orphans.length + missingFiles.length;
				if (totalChanges > 0) {
					vscode.window.showInformationMessage(
						`Code graph updated: ${orphans.length} deleted, ${missingFiles.length} added`
					);
				}

			} catch (error: any) {
				console.error('Reconciliation failed:', error);
				vscode.window.showErrorMessage(`Failed to validate code graph: ${error.message}`);
			} finally {
				this.isReconciling = false;
			}
		});
	}

	/**
	 * Find nodes in database where files no longer exist on disk
	 */
	private async findOrphanNodes(): Promise<string[]> {
		const orphans: string[] = [];

		// Query all FILE nodes from database
		const result = await this.dbService.query(
			'MATCH (f:FILE) RETURN f.id as id, f.path as path'
		);

		for (const row of result.data || []) {
			const filePath = row.path;

			// Check if file still exists on disk
			if (!fs.existsSync(filePath)) {
				orphans.push(row.id);
				console.log(`Orphan detected: ${filePath}`);
			}
		}

		return orphans;
	}

	/**
	 * Find files on disk that aren't in the database
	 */
	private async findMissingFiles(): Promise<string[]> {
		const missingFiles: string[] = [];

		// Get configured watch folders
		const config = vscode.workspace.getConfiguration('falkordb');
		const watchFoldersStr = config.get<string>('watchFolders', 'src');
		const watchFolders = watchFoldersStr.split(',').map(f => f.trim());

		// Get all files currently in database
		const dbResult = await this.dbService.query(
			'MATCH (f:FILE) RETURN f.path as path'
		);
		const dbPaths = new Set(dbResult.data?.map((row: any) => row.path) || []);

		// Scan each watch folder
		for (const folder of watchFolders) {
			const folderPath = path.join(this.workspaceRoot, folder);

			if (!fs.existsSync(folderPath)) {
				continue;
			}

			// Recursively find all files
			const filesOnDisk = this.scanDirectory(folderPath);

			// Find files not in database
			for (const filePath of filesOnDisk) {
				if (!dbPaths.has(filePath)) {
					missingFiles.push(filePath);
					console.log(`Missing file detected: ${filePath}`);
				}
			}
		}

		return missingFiles;
	}

	/**
	 * Recursively scan directory for all files
	 */
	private scanDirectory(dirPath: string): string[] {
		const files: string[] = [];

		try {
			const entries = fs.readdirSync(dirPath, { withFileTypes: true });

			for (const entry of entries) {
				// Skip hidden files and node_modules
				if (entry.name.startsWith('.') || entry.name === 'node_modules') {
					continue;
				}

				const fullPath = path.join(dirPath, entry.name);

				if (entry.isDirectory()) {
					// Recursively scan subdirectory
					files.push(...this.scanDirectory(fullPath));
				} else if (entry.isFile()) {
					files.push(fullPath);
				}
			}
		} catch (error) {
			console.error(`Error scanning directory ${dirPath}:`, error);
		}

		return files;
	}

	/**
	 * Delete orphaned nodes from database
	 */
	private async cleanupOrphans(nodeIds: string[]): Promise<void> {
		for (const nodeId of nodeIds) {
			await this.dbService.deleteNode(nodeId);
		}
	}

	/**
	 * Index files that exist on disk but not in database
	 */
	private async indexMissingFiles(filePaths: string[]): Promise<void> {
		// Import FileSystemIndexer to reuse indexing logic
		const { FileSystemIndexer } = await import('./FileSystemIndexer');
		const indexer = new FileSystemIndexer();

		// Index each missing file
		// (This is simplified - in production, batch these operations)
		for (const filePath of filePaths) {
			// TODO: Call indexer method to index single file
			// For now, just log
			console.log(`Would index: ${filePath}`);
		}
	}

	/**
	 * Verify that file metadata (size, timestamps) is accurate
	 */
	private async verifyFileMetadata(): Promise<void> {
		// Query all files
		const result = await this.dbService.query(
			'MATCH (f:FILE) RETURN f.id as id, f.path as path, f.size as size, f.modifiedAt as modifiedAt'
		);

		for (const row of result.data || []) {
			if (!fs.existsSync(row.path)) {
				continue; // Already handled by orphan detection
			}

			const stats = fs.statSync(row.path);

			// Check if metadata differs
			if (stats.size !== row.size || stats.mtimeMs !== row.modifiedAt) {
				// Update node with current metadata
				await this.dbService.updateNode(row.id, {
					size: stats.size,
					modifiedAt: stats.mtimeMs
				});
			}
		}
	}

	/**
	 * PHASE 3: Start periodic reconciliation
	 * Runs every 5 minutes if no manual changes detected
	 */
	public startPeriodicReconciliation(): void {
		// Check every minute, reconcile if needed
		this.reconciliationTimer = setInterval(() => {
			const timeSinceLastRecon = Date.now() - this.lastReconciliation;
			const FIVE_MINUTES = 5 * 60 * 1000;

			if (timeSinceLastRecon > FIVE_MINUTES) {
				console.log('Triggering periodic reconciliation');
				this.reconcileInBackground();
			}
		}, 60 * 1000); // Check every 1 minute
	}

	/**
	 * Stop periodic reconciliation
	 */
	public dispose(): void {
		if (this.reconciliationTimer) {
			clearInterval(this.reconciliationTimer);
		}
	}
}
```

### Step 3: Add Atomic Update Pattern

Create `src/services/AtomicUpdate.ts`:

```typescript
import { FalkorDBService } from './FalkorDBService';
import { GraphViewProvider } from '../providers/GraphViewProvider';

/**
 * Ensures database and view updates happen atomically
 */
export class AtomicUpdate {
	private dbService: FalkorDBService;
	private graphView?: GraphViewProvider;

	constructor(graphView?: GraphViewProvider) {
		this.dbService = FalkorDBService.getInstance();
		this.graphView = graphView;
	}

	/**
	 * Execute a database operation and update view atomically
	 * If either fails, rollback to previous state
	 */
	async execute<T>(
		operation: () => Promise<T>,
		rollback?: () => Promise<void>
	): Promise<T> {
		let result: T;

		try {
			// Step 1: Execute database operation
			result = await operation();

			// Step 2: Update view (if operation succeeded)
			if (this.graphView) {
				await this.graphView.refresh();
			}

			return result;
		} catch (error) {
			// If anything fails, attempt rollback
			if (rollback) {
				try {
					await rollback();
				} catch (rollbackError) {
					console.error('Rollback failed:', rollbackError);
				}
			}
			throw error;
		}
	}
}
```

**Usage Example**:

```typescript
const atomicUpdate = new AtomicUpdate(graphViewProvider);

// Create a file node atomically
await atomicUpdate.execute(
	// Operation: Create node
	async () => {
		await dbService.createNode(fileNode);
	},
	// Rollback: Delete node if view update fails
	async () => {
		await dbService.deleteNode(fileNode.id);
	}
);
```

## Reconciliation Strategies

### Strategy 1: Full Scan Reconciliation

**When to Use**: Small projects (< 1000 files), initial indexing, or manual refresh

**How It Works**:
1. Scan entire file system
2. Compare with all nodes in database
3. Find differences (added, deleted, modified)
4. Apply all changes

**Pros**:
- Simple to implement
- Guaranteed correct result
- No edge cases

**Cons**:
- Slow for large projects
- CPU intensive

**Code Example**:

```typescript
async function fullScanReconciliation(): Promise<void> {
	// 1. Get all files from database
	const dbNodes = await db.getAllFileNodes();
	const dbPathSet = new Set(dbNodes.map(n => n.path));

	// 2. Scan entire file system
	const fsFiles = await scanAllWatchedFolders();
	const fsPathSet = new Set(fsFiles.map(f => f.path));

	// 3. Find differences
	const toDelete = dbNodes.filter(n => !fsPathSet.has(n.path));
	const toAdd = fsFiles.filter(f => !dbPathSet.has(f.path));
	const toUpdate = fsFiles.filter(f => {
		const dbNode = dbNodes.find(n => n.path === f.path);
		return dbNode && (dbNode.size !== f.size || dbNode.modifiedAt !== f.modifiedAt);
	});

	// 4. Apply changes
	await deleteNodes(toDelete);
	await createNodes(toAdd);
	await updateNodes(toUpdate);
}
```

### Strategy 2: Incremental Reconciliation

**When to Use**: Medium projects (1k-10k files), periodic validation

**How It Works**:
1. Store last reconciliation timestamp in database
2. Only check files modified since that timestamp
3. Quick validation of recently changed files

**Pros**:
- Much faster than full scan
- Scales better

**Cons**:
- Relies on file system timestamps (can be unreliable)
- Might miss changes if timestamps are wrong

**Code Example**:

```typescript
async function incrementalReconciliation(): Promise<void> {
	// 1. Get last sync time from database
	const lastSync = await db.getMetadata('lastSyncTime') || 0;

	// 2. Find files modified since last sync
	const recentFiles = await findFilesModifiedSince(lastSync);

	// 3. Reconcile only those files
	for (const file of recentFiles) {
		const dbNode = await db.getNodeByPath(file.path);

		if (!dbNode) {
			// File added
			await db.createNode(createFileNode(file));
		} else if (file.size !== dbNode.size || file.modifiedAt !== dbNode.modifiedAt) {
			// File modified
			await db.updateNode(dbNode.id, {
				size: file.size,
				modifiedAt: file.modifiedAt
			});
		}
	}

	// 4. Check for deleted files (orphan detection)
	const orphans = await findOrphanNodes();
	await cleanupOrphans(orphans);

	// 5. Update sync timestamp
	await db.setMetadata('lastSyncTime', Date.now());
}
```

### Strategy 3: Content Hash-Based

**When to Use**: Critical accuracy needed, large files, or when timestamps are unreliable

**How It Works**:
1. Compute hash (SHA-256) of file contents
2. Store hash in database
3. Compare hashes to detect changes

**Pros**:
- Most reliable change detection
- Detects any content changes

**Cons**:
- CPU intensive (hashing large files)
- Slower than timestamp-based

**Code Example**:

```typescript
import * as crypto from 'crypto';

async function computeFileHash(filePath: string): Promise<string> {
	const content = fs.readFileSync(filePath);
	return crypto.createHash('sha256').update(content).digest('hex');
}

async function hashBasedReconciliation(): Promise<void> {
	const fileNodes = await db.getAllFileNodes();

	for (const node of fileNodes) {
		if (!fs.existsSync(node.path)) {
			// File deleted
			await db.deleteNode(node.id);
			continue;
		}

		// Compute current hash
		const currentHash = await computeFileHash(node.path);

		if (currentHash !== node.contentHash) {
			// File changed - re-index
			await reindexFile(node.path, currentHash);
		}
	}
}
```

### Strategy 4: Hybrid Approach (Recommended)

**Combine strategies based on project size and context**:

```typescript
async function smartReconciliation(): Promise<void> {
	const fileCount = await estimateFileCount();

	if (fileCount < 1000) {
		// Small project: Full scan is fast enough
		await fullScanReconciliation();
	} else if (fileCount < 10000) {
		// Medium project: Incremental + sampling
		await incrementalReconciliation();
		await sampleValidation(10); // Spot-check 10 random files
	} else {
		// Large project: Lazy + on-demand
		await lazyReconciliation();
	}
}

async function sampleValidation(count: number): Promise<void> {
	const allNodes = await db.getAllFileNodes();
	const samples = selectRandomSample(allNodes, count);

	for (const node of samples) {
		if (!fs.existsSync(node.path)) {
			console.warn('Sample validation found inconsistency, triggering full scan');
			await fullScanReconciliation();
			break;
		}
	}
}
```

## Edge Cases

### Edge Case 1: Symbolic Links

**Problem**: File system contains symlinks that might create duplicate nodes

**Solution**:
```typescript
function isSymlink(filePath: string): boolean {
	const stats = fs.lstatSync(filePath);
	return stats.isSymbolicLink();
}

// Skip symlinks during indexing
if (isSymlink(filePath)) {
	console.log(`Skipping symlink: ${filePath}`);
	continue;
}
```

### Edge Case 2: Case-Sensitive vs Case-Insensitive File Systems

**Problem**: `src/File.ts` and `src/file.ts` are different on Linux, same on Windows/Mac

**Solution**:
```typescript
// Normalize paths to handle case sensitivity
function normalizePath(filePath: string): string {
	// On case-insensitive systems, use lowercase
	if (process.platform === 'win32' || process.platform === 'darwin') {
		return filePath.toLowerCase();
	}
	return filePath;
}
```

### Edge Case 3: Very Large Files

**Problem**: Computing hashes for multi-GB files is slow

**Solution**:
```typescript
const MAX_HASH_SIZE = 10 * 1024 * 1024; // 10 MB

async function computeFileHash(filePath: string): Promise<string> {
	const stats = fs.statSync(filePath);

	if (stats.size > MAX_HASH_SIZE) {
		// For large files, use metadata-based fingerprint
		return `${stats.size}-${stats.mtimeMs}`;
	}

	// For small files, use content hash
	const content = fs.readFileSync(filePath);
	return crypto.createHash('sha256').update(content).digest('hex');
}
```

### Edge Case 4: Rapidly Changing Files

**Problem**: File changes multiple times during reconciliation

**Solution**:
```typescript
async function stableFileCheck(filePath: string): Promise<boolean> {
	// Check file stability: wait for changes to settle
	const stats1 = fs.statSync(filePath);
	await new Promise(resolve => setTimeout(resolve, 100)); // 100ms delay
	const stats2 = fs.statSync(filePath);

	// If size or mtime changed, file is still being modified
	return stats1.size === stats2.size && stats1.mtimeMs === stats2.mtimeMs;
}
```

## Performance Optimization

### 1. Batch Database Operations

**Problem**: Individual queries are slow

**Solution**:
```typescript
async function batchDeleteNodes(nodeIds: string[]): Promise<void> {
	// Instead of deleting one by one:
	// for (const id of nodeIds) await db.deleteNode(id);

	// Batch delete in single query:
	await db.query(
		'MATCH (n) WHERE n.id IN $ids DETACH DELETE n',
		{ ids: nodeIds }
	);
}
```

### 2. Parallel Processing

**Problem**: Sequential file scanning is slow

**Solution**:
```typescript
async function parallelScan(directories: string[]): Promise<string[]> {
	// Scan directories in parallel
	const results = await Promise.all(
		directories.map(dir => scanDirectory(dir))
	);

	// Flatten results
	return results.flat();
}
```

### 3. Debouncing Reconciliation

**Problem**: Too many reconciliations triggered close together

**Solution**:
```typescript
class DebouncedReconciler {
	private timeout?: NodeJS.Timeout;
	private readonly DEBOUNCE_MS = 5000; // 5 seconds

	scheduleReconciliation(): void {
		if (this.timeout) {
			clearTimeout(this.timeout);
		}

		this.timeout = setTimeout(() => {
			this.reconcileInBackground();
		}, this.DEBOUNCE_MS);
	}
}
```

### 4. Caching File System Queries

**Problem**: Repeated `fs.existsSync()` calls are slow

**Solution**:
```typescript
class FileExistenceCache {
	private cache = new Map<string, boolean>();
	private cacheExpiry = 5000; // 5 seconds

	exists(filePath: string): boolean {
		const cached = this.cache.get(filePath);
		if (cached !== undefined) {
			return cached;
		}

		const exists = fs.existsSync(filePath);
		this.cache.set(filePath, exists);

		// Clear cache after expiry
		setTimeout(() => this.cache.delete(filePath), this.cacheExpiry);

		return exists;
	}
}
```

### 5. Limit Reconciliation Scope

**Problem**: Reconciling entire workspace is unnecessary if only one folder changed

**Solution**:
```typescript
async function scopedReconciliation(changedFolder: string): Promise<void> {
	// Only reconcile the specific folder that changed
	const filesInFolder = await scanDirectory(changedFolder);

	// Reconcile just this subset
	await reconcileFiles(filesInFolder);
}
```

## Common Mistakes to Avoid

### ❌ Mistake 1: Blocking UI Thread

```typescript
// BAD: Blocks extension activation
export function activate() {
	synchronizer.reconcileInBackground(); // Runs synchronously!
	// Extension hangs until reconciliation completes
}

// GOOD: Run asynchronously
export function activate() {
	setTimeout(() => {
		synchronizer.reconcileInBackground();
	}, 2000); // Non-blocking
}
```

### ❌ Mistake 2: Not Handling Failed Updates

```typescript
// BAD: No error handling
await db.createNode(node);
graphView.refresh(); // Might fail, now DB and view are out of sync

// GOOD: Atomic update with rollback
try {
	await db.createNode(node);
	await graphView.refresh();
} catch (error) {
	await db.deleteNode(node.id); // Rollback
	throw error;
}
```

### ❌ Mistake 3: Ignoring Race Conditions

```typescript
// BAD: Multiple reconciliations can run simultaneously
async function reconcile() {
	const orphans = await findOrphans();
	await cleanupOrphans(orphans); // Another reconciliation might delete these first!
}

// GOOD: Guard with lock
if (this.isReconciling) return;
this.isReconciling = true;
try {
	// ... reconciliation logic
} finally {
	this.isReconciling = false;
}
```

### ❌ Mistake 4: Trusting File System Timestamps

```typescript
// BAD: Timestamps can be unreliable
if (file.mtimeMs > lastSync) {
	// File might have been edited before lastSync, but clock was wrong
}

// GOOD: Combine with content hashing for critical files
if (file.mtimeMs > lastSync || contentChanged(file)) {
	await reindex(file);
}
```

## Summary

**Key Principles**:
1. **Fast Startup**: Load cached data immediately (< 100ms)
2. **Background Validation**: Reconcile without blocking UI
3. **Atomic Updates**: Database and view always stay in sync
4. **Eventual Consistency**: Detect and fix drift over time
5. **User Feedback**: Show progress during reconciliation

**Core Pattern**: **Stale-While-Revalidate**
- Show data → Validate → Update if needed

**Three Phases**:
1. Load from database (immediate)
2. Reconcile in background (2 seconds after startup)
3. Periodic validation (every 5 minutes)

**Result**: Fast, reliable, self-healing code graph that remains synchronized even when external changes occur.
