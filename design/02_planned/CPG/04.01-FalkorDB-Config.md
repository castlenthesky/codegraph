**The VS Code extension ecosystem has clear, battle-tested conventions for handling external service dependencies like a graph database (FalkorDB in your case).** The goal is always minimal friction for users, non-invasive integration, and graceful degradation—never forcing manual edits to project files like `pyproject.toml` or `package.json` for runtime config.

### Do NOT ask users to edit pyproject.toml or package.json for connection config
This is **not** best practice and would feel out of place (and potentially break CI, linting, or team workflows). Those files are for build/dependency metadata.

- `pyproject.toml` / `package.json` are **not** where VS Code extensions store or read runtime configuration.
- Database-focused extensions (Neo4j for VS Code, Redis for VS Code, SQLTools, MongoDB tools, PostgreSQL, etc.) **never** do this. They use the official VS Code configuration API instead.
- If your extension targets Python/Node projects and users write code *against* FalkorDB, you *can* optionally offer a one-time helper command ("Add FalkorDB client dependency") that appends `falkordb` (Python) or `falkordb`/`falkordblite` (npm) to their dependency file and drops a sample `.env` or config snippet. But this is optional polish—**not** the primary mechanism.

### Use VS Code settings + a dedicated connection manager (the universal standard)
This is exactly how every major database extension works and what the ecosystem expects:

1. **Define settings in your `package.json`** (under `contributes.configuration`):
   ```json
   "configuration": {
     "title": "FalkorDB",
     "properties": {
       "falkordb.connectionMode": {
         "type": "string",
         "enum": ["embedded", "remote"],
         "default": "embedded",
         "description": "Whether to use the zero-config embedded FalkorDBLite or a remote instance"
       },
       "falkordb.host": { "type": "string", "default": "localhost" },
       "falkordb.port": { "type": "number", "default": 6379 },
       "falkordb.password": { "type": "string" }, // marked as secret in UI
       "falkordb.graphName": { "type": "string", "default": "default" },
       "falkordb.dataPath": { "type": "string", "default": "${workspaceFolder}/.falkordb" }
     }
   }
   ```
   Users can set these globally or per-workspace (stored in `.vscode/settings.json`—perfectly standard).

2. **Build a connection manager UI** (sidebar view or tree view in the Activity Bar):
   - Commands: `FalkorDB: Add Connection`, `FalkorDB: Switch Connection`, `FalkorDB: Test Connection`.
   - Store non-sensitive data in `context.globalState` / `context.workspaceState`.
   - Store passwords/tokens in `context.secrets` (VS Code's secure SecretStorage).
   - Show status in the status bar (e.g., "FalkorDB: embedded • connected" or "FalkorDB: localhost:6379").

3. **Onboarding wizard** on first activation (or when connection fails): Use `vscode.window.showInformationMessage` + quick picks / input boxes. This is extremely common and user-friendly.

This approach gives you workspace-scoped config, multi-connection support if needed, and full compatibility with settings sync, profiles, and Dev Containers.

### Fallback: Yes—use FalkorDBLite embedded as the default zero-config mode
This is **the killer recommendation** and perfectly matches what you described ("in memory + persist to a small file in the project").

FalkorDB officially ships **FalkorDBLite** (npm: `falkordblite`—since your extension is Node.js/TypeScript).

- Zero-config: `import { FalkorDB } from 'falkordblite'; const db = await FalkorDB.open();`
- It automatically starts an embedded Redis + FalkorDB module as a sub-process (Unix socket, no open ports).
- Persistence: Pass a path → `FalkorDB.open({ path: context.storageUri?.fsPath || '${workspaceFolder}/.falkordb/data' })`. Data survives restarts and is stored in a small file/folder.
- Lifecycle: Start on extension activation (if embedded mode), cleanly shut down on `deactivate()`.
- Add `.falkordb/` to `.gitignore` recommendation in your README/onboarding.

**Why this is best practice**:
- Matches exactly how lightweight embedded DBs are handled in VS Code extensions (e.g., SQLite-based tools, local vector stores in AI extensions).
- No Docker or external server required for 90% of users → huge UX win.
- You can still let users switch to "remote" mode (or Docker) via a setting/command.
- Optional helper command: "Start Local FalkorDB (Docker)" that runs `docker run -d -p 6379:6379 -v ${workspaceFolder}/.falkordb:/var/lib/falkordb/data falkordb/falkordb` (very common pattern).

### Bonus ecosystem conventions to make your extension feel native
- **Dev Containers support**: Ship a sample `.devcontainer/devcontainer.json` with a FalkorDB service (or just document it). Many extensions do this.
- **Error handling & status**: Always show clear, actionable messages ("Could not connect—falling back to embedded mode" with a "Configure" button).
- **Performance**: FalkorDBLite is designed for exactly this (local dev, prototyping, per-project data). It's fast and lightweight.
- **Migration path**: Easy to switch from embedded → remote later (just change the connection).
